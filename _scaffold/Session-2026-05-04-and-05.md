### #2 — Fix disappearing 7-day plan after `Generate plan`  ✅
**The bug.** `/lead/[id]` Generate plan → plan rendered → ~30s later the plan vanished, "No plan yet" empty state returned. If you clicked Approve & run fast enough, it persisted.

**Root cause.** Two compounding bugs:
1. `LeadToolbar.tsx` mounts `<AutoRefresh intervalMs={30000} />` which re-runs the Server Component every 30s.
2. `load-lead-box.ts:76` had `getLatestPlanForLead(leadId).catch(() => null)` — any transient Supabase read error silently became `plan: null`, indistinguishable from "no plan exists."

**Fix.**
- `AutoRefresh` now defaults to OFF on `/lead/[id]` (auto-refresh pill reads "auto off"). Manual refresh button + tab-focus refresh still work.
- `load-lead-box.ts` replaces the swallowed catch with try/catch + `console.error("[load-lead-box] ...")` + a new `planError` field surfaced through to the UI.
- `PlanSection.tsx` gained a "Plan fetch failed" card distinct from "No plan yet" — shows the error + a Refresh link.

Files: `src/app/lead/[id]/AutoRefresh.tsx`, `load-lead-box.ts`, `PlanSection.tsx`, `page.tsx`.

### #4 — Fold MCP-fallback failures into recovered groups  ✅
**The bug.** When the LLM tried an API tool, it failed (e.g. `Search "Holloway events"` hit a `klausApiError`), the LLM fell back to a different API tool (`list_leads`) which succeeded — but the chat panel still rendered the failed Search call with a giant red FAILED badge. Successful chains looked broken.

**Fix.** New `src/lib/tool-groups.ts` with pure `groupToolTrace(calls)` function. Walks the trace, accumulates failures into a buffer, flushes as a "recovered" group when a success follows. Server-side: `/api/chat/route.ts` emits a new `cmk:tool-groups` JSON fence (replaces `cmk:tools` — old fence still parsed for backward-compat with persisted history). Client: new `ToolGroupPanel` renders three shapes (single-success, recovered with collapsed prior attempts, failed). Initial heuristic was too tight (only `close_mcp_call` counted as recovery); loosened to "any next-call success after failure(s)" because the LLM also recovers via dropping to simpler API tools.

### Phase 5 — Comeketo-vocabulary node taxonomy  (groundwork, ~half day)

**Why.** Currently `WorkflowManifest` node `kind` is opaque — for plans it's `"plan_day"`, for automation_drafts it's whatever the original Workflow had (cron, llm_call, slack_post, etc.). The graph visualization and inspector know nothing about *what each node actually does*. To enable Phases 6–8 (shape library, dispatcher, authoring), the node kind needs to speak a real vocabulary.

**The vocabulary** (per Jake's 2nd-AI analysis, I agree this is the right knife edge):

```
TRIGGER                  ACTION                      GUARD
─────────                ──────                      ─────
schedule.daily           plan.generate (n-day)       reply_gate.check
schedule.cron            plan.regenerate             frequency_cap.enforce
event.lead.replied       touch.compose               nepq_voice.lint
event.lead.stale         touch.send                  approval.required
event.tasting.completed  task.create                 stop_signal.check
event.proposal.sent      activity.log                ownership.gate
manual.operator          slack.notify                snapshot.match
                         heartbeat.skip
                         box.update
```

These are the nouns the runtime already supports — they're just not first-class in the manifest yet.

**What to build.**
1. Define `NodeKind` union type in `workflow-manifest.ts` covering all of the above.
2. Update `leadPlanToManifest` to emit *per-touch* nodes per day instead of one node per day. So a day with [SMS, email, task] becomes 3 nodes: `touch.send` (sms) → `touch.send` (email) → `task.create`. Day grouping becomes a visual cluster, not a single node.
3. Update `PlanGraphView` to lay out per-touch nodes in vertical stacks per day, with horizontal day-to-day flow (so you get a 2D grid: days across, touches within day).
4. Inspector shows the typed node config based on kind.

**Why this matters.** Once node kinds are first-class, the simulator can show per-node verdicts (currently aggregates per day), the Mermaid output reads as actual workflow logic, and Phases 6–8 have something to compose.

**Files.** `src/lib/workflow-manifest.ts` (NodeKind union + adapter rewrite), `src/app/lead/[id]/PlanGraphView.tsx` (2D layout), `src/components/AutomationCanvas.tsx` (maybe — depending on whether the layout fits the existing endpoint geometry or needs new edge routing).

**Risk.** Per-touch nodes mean ~3-4x more nodes on the canvas. Need to make sure the existing SVG renderer + edge routing handles vertical stacks. May need a "day cluster" container (a rect grouping the touches of one day) — that's new geometry.

### Phase 6 — Workflow shape library + multi-shape heartbeat dispatcher  (the big one, ~3-5 days)

**The vision.** Today every plan is a 7-day NEPQ cycle because that's the only shape `generateSevenDayPlanForLead` knows. Tomorrow Andre has 5-10 named shapes:

```
shape 1: 7-day NEPQ cycle           ← already shipped
shape 2: reschedule listener        ← reactive (event.lead.replied → ...)
shape 3: post-tasting follow-up     ← event-triggered
shape 4: cold-revival sequence      ← scheduled (lead stale 30+ days)
shape 5: ghosted lead recovery      ← stale-detector triggered
shape 6: Andre's personal whatever  ← custom
```

Each shape is a `WorkflowManifest`. The heartbeat sweep walks active shapes, evaluates triggers against current lead state, and instantiates plans from matching shapes.

**What to build.**
1. **`workflow_shapes` table** in Supabase. Columns: `id`, `name`, `manifest_json`, `status` (draft/active/archived), `applies_to` (lead_status filter, etc.), `created_at`, `updated_at`.
2. **Shape persistence layer** in `src/lib/workflow-shapes.ts` — CRUD parallel to plans-db.ts.
3. **Trigger evaluator** in heartbeat — for each active shape with an event-style trigger, check: did this trigger fire for any lead since last sweep? Spawn a plan from the shape's template if so.
4. **Plan-from-shape codegen** — take a shape's manifest + a lead context → produce a concrete `lead_plans` row. The shape provides the template, the lead provides the variables.
5. **Shape library UI** (or just a CLI for v1 — Andre might not need a UI immediately) — list shapes, view manifest, activate/deactivate.

**Two big architectural decisions to make before starting.**

**Q1: Triggers — webhook-driven or poll-driven?**
- Close has webhooks (already partially wired — `close_webhook_events` table exists). Webhook-driven is real-time but requires shape→trigger routing per webhook event.
- Polling is simpler — heartbeat scans every N minutes, evaluates all triggers, spawns plans. Latency is the tradeoff.
- Hybrid: schedule-style triggers (cron) run on heartbeat tick; event-style triggers (`lead.replied`, `tasting.completed`) run on webhook intake.

**Q2: Plan instantiation — eager or lazy?**
- Eager: trigger fires → plan row created in `lead_plans` immediately → next heartbeat picks it up like any plan.
- Lazy: trigger fires → "shape match" recorded → heartbeat materializes the plan inline at fire time.
- Eager is simpler, gives the operator a reviewable artifact (plan card with days), but creates churn in `lead_plans`.

My read: hybrid triggers + eager instantiation. Get something visible fast, defer streaming.

**Files.** New `src/lib/workflow-shapes.ts`. New Supabase migration. Heartbeat additions in `src/lib/heartbeat.ts`. Maybe a new `/shapes` route + UI later.

**Skip-if.** Andre doesn't have a clear second shape he wants yet. Don't build a multi-shape system speculatively — wait until there's a real second shape worth coding.

## Part 5 — Coding rules + idioms learned this session

For the next agent's benefit:

- **CLAUDE.md three-ledger rule is non-negotiable.** Every move taps Goals/Problems/Global with explicit BEFORE and AFTER announcements to Jake. One pair per phase, not per micro-task.
- **Jake's CLAUDE.md (global) deprioritizes security ceremony.** Don't add gates, locks, or "best practices" posture unless asked. Ship-and-iterate over harden-then-ship.
- **No flattery.** Acknowledge wins briefly when warranted, but don't gush. Push back when wrong.
- **Don't push to main on every change.** Local commits are fine; Vercel auto-deploys on push. Always confirm before pushing.
- **`npm run fresh` after non-trivial sprints.** Next 14's dev server caches webpack chunks; new Server Actions / new dynamic routes / shuffled client-server boundaries desync. The script nukes `.next` and restarts.
- **Land in position.** Don't end a sprint by asking "what's next?" — pick the next move from the Goals milestone + Watch items, execute or tee up. Hand the choice back only when there's a real fork.
- **Atomize hard moves before executing.** If a move touches multiple files / multiple systems, write the atomization first.
- **Plan mode for big architectural sprints; direct execution for small surgical fixes.** Jake explicitly toggles plan mode when he wants deliberation.
- **Auto mode means execute without asking for routine decisions** but still confirm destructive actions.
- **Stale `.next/types/` errors after deleting routes** — `rm -rf .next` fixes them. Standard step after route deletions.
- **The Close API quirks worth knowing**: `_limit` caps at 200 on `/lead/`, search uses plain text via `GET /lead/?query=` not `POST /data/search/` (Klaus DSL is broken), workflow params are `sequence_id` everywhere even though Close UI calls them workflows.
- **System prompt size matters but not as much as you'd think.** ~280 lines (~5000 tokens) of instructions + tool catalog is fine; the model handles it.
